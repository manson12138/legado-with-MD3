# M0：项目决策与迁移账本

## AI 执行规则（必须先读）

1. 开始本阶段前必须完整阅读上级文档 `../FLUTTER_REWRITE_EXECUTION_PLAN.md`。
2. 本阶段只建立事实、规则、映射和验收基线，不创建 Flutter 业务代码。
3. 所有结论必须能追溯到用户确认、Android 源文件或数据库 Schema；不得凭印象填写。
4. 不得把“暂不迁移”写成“无需迁移”；所有延期功能都必须留在功能矩阵中。
5. 不得修改原 Android 业务代码，不得运行编译、测试、Lint、格式化或静态检查。
6. 所有新增文档字段必须含义明确，状态只能使用总方案规定的固定枚举。
7. 发现冲突、缺失入口或无法判断的平台能力时，记录问题并向用户确认，不得静默决定。
8. 本阶段未通过退出门禁前，AI 不得宣布 M0 完成，也不得进入 M1。

## 阶段目标

建立整个 Flutter 重写项目的唯一事实来源，使后续 AI 能回答：要迁移什么、对应原文件在哪里、由哪一层负责、当前处于什么状态、如何证明完成。

## 本阶段不包含

- 不创建 `flutter_app`。
- 不选择具体数据库、路由或状态管理依赖。
- 不实现书源解析。
- 不绘制业务页面。
- 不编写 Kotlin/Swift 平台桥接。

## 前置条件

- 用户已经确认同仓库、独立 App、不迁移旧数据、Android 优先、iOS 自用安装。
- 用户已经确认普通规则和 JavaScript 都必须兼容。
- 原 Android 项目作为冻结功能基线。

## 需要建立的文档

1. 项目决策记录：确认项、待确认项、修改日期、修改原因。
2. 架构边界：UI、状态、领域、数据、规则引擎、平台桥接职责。
3. 文件映射表：Android 路径、Flutter 路径、类型、方法、状态和差异。
4. 功能矩阵：入口、主要操作、数据、Android 状态、iOS 状态。
5. UI 规范：统一视觉、Design Token、公共组件、页面状态和阅读器规则。
6. JavaScript 兼容规范：L1/L2/L3 范围、样本、错误模型和不可兼容边界。
7. 平台能力矩阵：纯 Dart、插件、Kotlin、Swift、替代或不支持。
8. 验收规范：每个阶段和 Feature 的验证模板。

## 执行步骤

### 1. 冻结基线信息

- 记录当前仓库路径、基线分支和基线提交，由用户确认是否长期固定。
- 记录 Android applicationId、最低/目标系统版本和最新数据库 Schema。
- 记录主模块、Rhino 模块、书籍解析模块和 Web 前端的职责。

### 2. 建立顶层包清单

- 盘点 `api`、`base`、`constant`、`data`、`domain`、`help`、`lib`、`model`、`receiver`、`service`、`ui`、`utils`、`web`。
- 每个包记录文件数量、主要职责、第一批是否需要、预期 Flutter 目录。

### 3. 建立功能入口清单

- 盘点 Manifest Activity、Service、Receiver、Intent filter 和权限。
- 盘点 MainActivity 路由及所有 Compose/legacy 页面入口。
- 对每个入口记录输入、输出、返回行为、系统依赖和数据依赖。

### 4. 建立首批详细映射

- 优先完整映射书源、规则、搜索、详情、目录、书架和阅读器。
- 找出关联的 XML、Adapter、Dialog、ViewModel、Entity、DAO、Repository、UseCase、Service 和工具类。
- 标记纯 UI 代码、可复用业务代码和 Android 专属代码。

### 5. 建立状态与风险登记

- 状态只使用 `NOT_STARTED`、`MAPPING`、`IN_PROGRESS`、`ANDROID_READY`、`IOS_READY`、`DONE`、`BLOCKED`、`NOT_APPLICABLE`。
- 风险至少包含 JavaScript/Rhino、WebView、Cookie、字符集、阅读器排版、后台任务和 iOS 平台限制。

## 注意点

- 数据库实际 Schema 文件已经高于旧文档描述，必须以仓库最新 Schema 为事实来源。
- Compose 页面与 legacy View 页面并存，不能只盘点 XML 或只盘点 Compose。
- 原 Activity 可能只是入口，真实业务可能在 ViewModel、单例 Model 或 Service 中。
- 某些工具类只为 Android Framework 服务，应映射为平台接口而不是机械翻译。
- “第一批不做”仍要登记未来阶段，防止永久遗失。

## 完成标准

只有同时满足以下条件，才说明 M0 已完成：

1. 第一批闭环中的每个 Android 入口都能找到 Flutter 预期归属。
2. 第一批关联的 Entity、DAO、规则、工具和平台能力没有“未知归属”。
3. 每个延期功能都保留在功能矩阵中。
4. 所有关键架构选择有明确决策或明确待确认项。
5. 下一阶段创建工程所需的目录名、应用名、applicationId、Bundle Identifier、系统最低版本已有确定值；若未确定，必须得到用户许可使用临时值。
6. 用户可以仅查看迁移账本就理解当前范围和完成定义。

## 阶段退出门禁

- 不存在首批功能的 `MAPPING` 未完成项。
- 不存在会改变工程标识或核心架构但尚未询问用户的问题。
- 文档之间没有相互矛盾的系统版本、范围或命名规则。
- AI 明确报告“文档基线完成，尚未创建 Flutter 工程”。
