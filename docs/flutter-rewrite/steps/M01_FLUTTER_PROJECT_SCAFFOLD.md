# M1：Flutter 工程骨架

## AI 执行规则（必须先读）

1. 必须先确认 M0 已通过退出门禁，并完整阅读总方案与 M0 文档。
2. 本阶段目标是建立可扩展工程骨架，不得提前实现书源、搜索、书架或阅读器业务。
3. 必须使用用户确认的目录名、应用名、applicationId 和 Bundle Identifier；不得把临时值当最终值。
4. Dart 优先；只有宿主初始化确实需要时才编辑 Kotlin/Swift。
5. 所有新增类、方法和变量必须有详细中文注释；禁止 Kotlin `!!` 和 Dart 强制空值断言 `!`。
6. 不运行 `flutter create` 以外的检查类命令；如果创建工程本身会隐式执行依赖解析，应提前说明。用户未授权下载时不得联网安装依赖。
7. 不运行 `flutter analyze`、测试、构建或真机启动，由用户执行。
8. 本阶段未通过退出门禁前不得开始 M2。

## 阶段目标

创建一个与原 Android App 共存的 Flutter Android/iOS 工程，建立统一主题、路由、依赖注入、错误边界、日志接口和 MVI/UDF 基础约定。

## 前置条件

- M0 完成。
- 工程标识已确认。
- Flutter stable 版本策略已确认并固定。
- Android `minSdk 26`、iOS `16.0` 若有变化已记录。

## 本阶段不包含

- 不建立完整业务数据库。
- 不实现 HTTP 和规则解析。
- 不实现真实书源导入。
- 不实现核心业务页面。
- 不复用原 Android 私有数据。

## 执行步骤

### 1. 创建工程

- 在仓库内创建 `flutter_app` 或用户确认的目录。
- 只启用 Android 和 iOS 平台。
- 设置独立 applicationId 和 Bundle Identifier，确保可与原 App 共存。
- 固定 Flutter/Dart SDK 版本并提交锁文件策略。

### 2. 建立目录

- 创建 `lib/src/app`、`data`、`domain`、`help`、`model`、`platform`、`service`、`ui`、`utils`。
- 创建 `packages` 作为未来自研平台插件位置，但不创建无用插件空壳。
- 创建测试目录，但不编写与当前骨架无关的占位测试。

### 3. 应用组合根

- `main.dart` 只负责初始化和启动应用。
- 应用根负责注入依赖、主题、路由和全局错误边界。
- 平台实现的选择只能发生在组合根或插件注册层。

### 4. UI 基础

- 定义颜色、字体、间距、圆角、阴影和动画 Design Token。
- 提供最小 `AppScaffold`、加载、空状态和错误状态组件。
- 创建欢迎页或占位主页，只验证结构，不模拟业务数据。
- 处理 Android edge-to-edge 与 iOS Safe Area 的统一入口。

### 5. MVI/UDF 基础

- 定义 State、Intent、Effect 的命名和职责示例。
- 定义 ViewModel/Controller 生命周期和依赖注入方式。
- 一次性导航和系统行为走 Effect，不存入长期状态。

### 6. 基础设施接口

- 建立日志抽象、应用错误类型和结果模型。
- 建立平台能力接口目录和命名规则，但不提前实现未来能力。
- 确保 UI 不直接依赖平台实现。

## 注意点

- 依赖注入、路由、数据库等库一旦选定会影响全项目，新增前必须记录选择理由。
- 不要为了“以后可能需要”引入大型框架。
- Flutter Android 宿主包名必须与原 App 不同。
- iOS 自用安装仍需要正确 Bundle Identifier 和签名目标。
- Debug 本地网络权限与 Release 权限必须区分。
- 统一 Flutter UI 不代表忽略系统返回、键盘和安全区域。

## 完成标准

1. 目录职责与总架构一致，没有把业务写入 `main.dart`。
2. Android/iOS 标识均为新 App 标识。
3. 页面能够从统一主题、路由和依赖容器启动。
4. 示例页面只包含结构验证，不包含虚假业务完成度。
5. Android edge-to-edge 与 iOS Safe Area 有统一处理策略。
6. 用户拥有明确的依赖获取、运行和人工查看步骤。
7. 用户运行后确认 Android 至少达到 A0；未得到用户结果前只能标记“代码待验证”。
