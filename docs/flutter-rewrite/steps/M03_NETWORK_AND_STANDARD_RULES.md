## 阶段目标

建立 Android/iOS 共用的 HTTP 与普通规则解析基础，能够在不依赖 JavaScript 的书源上完成搜索、详情、目录和正文解析。

## 执行步骤

### 1. 网络行为盘点

- 盘点请求方法、Header、Body、代理、DNS、重定向、Cookie、超时、缓存、字符集和压缩。
- 盘点 `AnalyzeUrl` 中请求选项和特殊 URL 语法。
- 盘点 WebView 请求何时介入，但本阶段只定义接口边界。

### 2. 统一 HTTP 接口

- 定义 Request、Response、Header、Body 和取消令牌。
- 统一 GET、POST 和必要请求类型。
- 响应保留原始字节，在明确字符集后转换文本。
- 错误至少区分取消、DNS、连接、TLS、超时、HTTP 状态、解码和未知错误。

### 3. Cookie 与会话

- 定义 Cookie 持久化和按域匹配规则。
- 明确普通 HTTP、JavaScript 与未来 WebView 的同步接口。
- 登录书源需要独立或共享会话时，行为必须可配置并与 Android 对照。

### 4. 普通规则解析器

- 实现 JSONPath、XPath、CSS/JSoup 等价选择器、Regex 和字符串规则。
- 保留规则链、取值、过滤、拼接、替换和列表/单值语义。
- 将解析器错误与“没有匹配结果”区分。
- 规则解析器不得直接显示 UI 错误。

### 5. 四段业务链路

- 搜索列表。
- 书籍详情。
- 章节目录。
- 章节正文。

每段建立固定输入、Android 输出和 Flutter 输出对照样本。

### 6. 性能与隔离

- 大文本、复杂正则和 HTML 解析不得阻塞 UI isolate。
- 网络请求支持取消和总超时。
- 多书源并发上限由上层控制，网络层提供基础取消能力。

## 注意点

- Dart 正则、Java 正则和 JavaScript 正则可能存在语法或 Unicode 差异。
- JSoup 与 Flutter HTML 库的选择器行为可能不同，必须用结果样本确认。
- 响应 Header 指定编码、HTML meta 编码和规则指定编码的优先级要与 Android 对照。
- 重定向后的最终 URL 可能参与相对地址解析。
- 空匹配、解析错误和网络错误不能都返回空列表。
- WebView 是受控兜底，不得成为所有请求的默认实现。
