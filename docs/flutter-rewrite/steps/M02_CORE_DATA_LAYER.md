# M2：核心数据层

## AI 执行规则（必须先读）

1. 必须确认 M1 已完成或用户明确允许继续，并阅读总方案、M0、M1 文档。
2. 本阶段只实现第一批闭环需要的数据模型、DAO、Repository、Gateway 和 UseCase。
3. 必须先读取 Android Entity、DAO、最新 Room Schema 和真实调用点，再定义 Dart 字段。
4. Flutter 数据库是全新数据库，不实现旧 App Room 迁移，但字段语义不得随意改变。
5. UI 不得直接访问 DAO；DAO 不得包含页面状态逻辑。
6. 不得用空字符串或 0 粗暴替代有业务意义的 `null`。
7. 所有新增类、方法和变量必须有详细中文注释；禁止 Dart `!`。
8. 不运行数据库测试、代码生成检查、analyze 或 build，由用户运行。

## 阶段目标

为书源、搜索、书架和阅读建立稳定的数据基础，使后续规则和 UI 都依赖明确的数据契约，而不是动态 Map 或临时模型。

## 执行步骤

### 1. 字段盘点

- 对照 Android Entity 和最新 Schema，记录字段名、类型、可空性、默认值、主键、索引和序列化名。
- 找出字段真实调用点，确认空值、空字符串、0 和缺失字段的差异。
- 标记 URL 主键、时间单位、排序字段和状态枚举。

### 2. Dart Entity

- 文件名使用 snake_case，类名保持 Android 领域名。
- 模型对外不可变，修改通过明确的 copy 方法或构造新实例。
- JSON 导入模型与数据库 Entity 如语义不同，应分开定义，不让外部动态数据污染数据库对象。
- 每个字段注释其 Android 对应字段和业务含义。

### 3. 数据库 Schema

- 建立全新数据库版本 1。
- 保留必要表名和字段名以便搜索对照。
- 明确定义主键、复合键、唯一约束和索引。
- 事务覆盖批量导入、书架关键写入和关联删除。

### 4. DAO

- DAO 只表达查询和写入。
- 查询名称尽量对应 Android Dao 方法。
- 流式观察使用明确 Stream；一次查询使用 Future。
- 不在 DAO 中进行网络、规则解析或 UI 格式化。

### 5. Repository/Gateway

- Repository 组合 DAO 和其他数据源。
- Domain 依赖 Gateway 抽象，不依赖数据库具体库。
- 错误转换为明确领域错误，不能把数据库异常直接传到 UI。

### 6. UseCase

- 只为具有业务含义、会复用或需要事务协调的动作创建 UseCase。
- 不为每个 DAO 方法机械创建一层同名空转 UseCase。
- 第一批至少覆盖书源导入、加入书架、读取目录、保存/恢复进度等边界。

## 注意点

- Android 文档中的数据库版本可能过时，必须以最新 Schema 文件为准。
- `Book`、`SearchBook` 和 `BaseBook` 的字段相似但职责不同，不能随意合并。
- Book URL、Chapter URL、Source URL 可能参与主键，规范化会影响唯一性。
- 时间戳必须统一毫秒或明确转换点。
- 新 App 不需要旧迁移，不代表可以忽略导入 JSON 的历史字段兼容。
- Repository 不应返回数据库库专属对象。
